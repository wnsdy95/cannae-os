# SOP Library

## 0. 목적

이 문서는 군대식 표준작전절차(SOP)를 LLM 운용 절차로 번역한 라이브러리다.

SOP의 목적은 에이전트를 기계적으로 묶는 것이 아니다. 반복 가능한 작업에서 매번 새로 판단할 필요가 없는 부분을 표준화하고, 판단이 필요한 부분만 지휘관 또는 상위 에이전트에게 올리기 위한 장치다.

LLM 운용에서 SOP는 다음 문제를 줄인다.

- 사용자 의도 누락.
- 하위 에이전트의 임의 해석.
- 출처 없는 주장.
- 검증되지 않은 산출물.
- 변경사항의 무단 확산.
- 반복 실패의 학습 누락.

핵심 원칙:

```text
SOP는 에이전트의 자유를 없애는 문서가 아니라,
자율 판단이 가능한 경계선을 명확히 하는 문서다.
```

## 1. SOP 표준 양식

모든 SOP는 아래 항목을 가진다.

| 항목 | 설명 | LLM 적용 |
| --- | --- | --- |
| 목적 | 이 절차가 방지하거나 달성하려는 것 | 작업의 성공 조건 |
| 적용 범위 | 언제 이 절차를 쓰는가 | 트리거 조건 |
| 역할 | 누가 무엇을 맡는가 | Commander, S2, S3, S4, Red Team 등 |
| 입력 | 시작 전에 필요한 정보 | 사용자 요청, 파일, 출처, 제약 |
| 절차 | 수행 순서 | 프롬프트 체인 또는 에이전트 워크플로 |
| 산출물 | 반드시 남겨야 하는 결과 | 문서, 코드, 검증 로그, 리스크 목록 |
| 승인 기준 | 언제 상위 승인이 필요한가 | 인간 승인 또는 상위 에이전트 승인 |
| 보고 기준 | 언제 보고해야 하는가 | CCIR, PIR, FFIR, EEFI |
| 검증 기준 | 완료 판정 방법 | 테스트, 근거 확인, 리뷰 |
| 기록 | 사후관리 자료 | AAR, decision log, source map |

## 2. SOP 작성 규칙

좋은 SOP는 짧고 실행 가능해야 한다.

나쁜 SOP:

```text
자료를 충분히 조사하고 품질 높은 답변을 작성한다.
```

좋은 SOP:

```text
1. 사용자 요구를 mission, intent, constraints로 분해한다.
2. 불확실하거나 최신성이 필요한 사실은 출처를 확인한다.
3. 출처별로 주장, 한계, LLM 적용점을 기록한다.
4. 산출물에 출처 링크와 적용 해석을 남긴다.
5. 해결되지 않은 질문은 PIR 목록에 추가한다.
```

SOP는 다음 표현을 피한다.

- 충분히.
- 적절히.
- 잘.
- 필요하면.
- 상황에 따라.

대신 다음처럼 쓴다.

- 최소 3개 이상의 1차 출처를 확인한다.
- 변경 파일을 모두 `rg --files`와 `wc -l`로 검증한다.
- 출처 없는 주장은 "가설"로 표시한다.
- 위험도가 high 이상이면 승인 요청으로 전환한다.

## 3. SOP-01 Research / Intelligence Collection

### 목적

리서치 에이전트가 환각 없이 근거 기반 자료를 수집하고, 그 자료를 LLM 운용 프레임워크로 번역하게 한다.

### 적용 범위

- 군사 교리, 조직 운영, 문서 체계, AI 운용법을 조사할 때.
- 사용자가 "deep research", "논문", "문서", "모든 자료"를 요구할 때.
- 최신성 또는 출처 정확성이 중요한 경우.

### 역할

| 역할 | 책임 |
| --- | --- |
| Commander | 리서치 목적, 범위, 우선순위 결정 |
| S2 Intelligence | 출처 수집, 신뢰도 평가, 불확실성 표시 |
| S3 Operations | 자료를 실행 가능한 프레임워크로 변환 |
| S5 Plans | 장기 프레임워크와 개념 연결 |
| Red Team | 과장, 비약, 출처 오류 검토 |

### 입력

- 사용자 목표.
- 현재 문서 세트.
- 검색 키워드.
- 이미 확보한 출처 목록.
- 답해야 할 PIR.

### 절차

1. 사용자 요청을 research mission으로 재작성한다.
2. 질문을 PIR, FFIR, EEFI로 나눈다.
3. 1차 출처를 우선 검색한다.
4. 정부/군 공식 교리, 학술 논문, 핸드북, 현장 보고서 순서로 신뢰도를 부여한다.
5. 각 출처마다 핵심 개념, 원문 맥락, LLM 적용점을 분리 기록한다.
6. 서로 충돌하는 내용은 하나로 합치지 말고 차이를 남긴다.
7. 아직 확인하지 못한 주장은 추정으로 표시한다.
8. 산출물을 `research-compendium.md` 또는 `source-map.md`에 반영한다.

### 승인 기준

다음 경우 Commander 승인 또는 사용자 확인이 필요하다.

- 공개 자료가 아닌 문서를 쓰려는 경우.
- 법률, 보안, 군사 작전의 실제 적용 조언으로 넘어가는 경우.
- 출처 간 충돌이 있는데 하나를 표준으로 삼아야 하는 경우.
- 프레임워크 핵심 용어를 바꾸는 경우.

### 산출물

- 출처 목록.
- 출처별 핵심 내용.
- LLM 적용 해석.
- 남은 연구 질문.
- source map 갱신.

## 4. SOP-02 Document Production

### 목적

리서치와 판단을 일회성 답변으로 흘려보내지 않고, 재사용 가능한 문서 체계로 축적한다.

### 적용 범위

- 프레임워크 문서 작성.
- 템플릿 작성.
- 정책, SOP, authority matrix 작성.
- 기존 문서 업데이트.

### 역할

| 역할 | 책임 |
| --- | --- |
| Commander | 문서의 목적과 독자 결정 |
| S3 Operations | 구조 설계와 실행 절차화 |
| S2 Intelligence | 근거 연결 |
| S6 Knowledge | 문서 위치, 링크, 색인 관리 |
| Red Team | 빈 주장, 중복, 용어 혼선 점검 |

### 절차

1. 새 문서가 필요한지, 기존 문서에 추가할지 결정한다.
2. 문서 목적을 첫 10줄 안에 적는다.
3. 독자가 바로 실행할 수 있는 순서로 섹션을 배치한다.
4. 개념 설명과 실제 템플릿을 분리한다.
5. 문서 끝에 관련 문서와 다음 작업을 남긴다.
6. README 또는 상위 문서에 링크를 추가한다.
7. `rg`로 새 문서가 색인에 걸리는지 확인한다.

### 산출물 기준

문서는 다음 조건을 만족해야 한다.

- 파일명만 보고 용도를 알 수 있다.
- 첫 섹션에서 목적을 알 수 있다.
- 표, 체크리스트, 절차 중 최소 하나가 있다.
- 관련 문서로 이동할 수 있다.
- 다음 작업자가 문맥 없이 이어갈 수 있다.

## 5. SOP-03 OPORD Prompt Generation

### 목적

사용자 요청을 군 작전명령(OPORD)식 프롬프트로 변환하여 하위 에이전트가 왜곡 없이 실행하게 한다.

### 적용 범위

- 복잡한 LLM 작업.
- 여러 에이전트가 참여하는 작업.
- 산출물 품질과 검증이 중요한 작업.
- 사용자의 의도가 길거나 모호한 작업.

### 절차

1. 사용자 요청에서 mission을 한 문장으로 뽑는다.
2. commander's intent를 목적, 핵심 성공 조건, 실패 방지 조건으로 나눈다.
3. situation에 현재 상태, 파일, 제약, 알려진 리스크를 적는다.
4. execution에 단계별 작업을 쓴다.
5. coordinating instructions에 금지사항, 보고 기준, 검증 기준을 적는다.
6. sustainment에 사용 가능한 도구, 시간, 토큰, 출처를 적는다.
7. command and signal에 보고 형식, 승인권, 중단 조건을 적는다.
8. backbrief를 요구한다.

### 최소 프롬프트 골격

```text
Mission:
Intent:
Situation:
Execution:
Constraints:
Authority:
CCIR:
Verification:
Deliverable:
Backbrief:
```

### 승인 기준

다음 경우 OPORD를 바로 실행하지 말고 backbrief를 먼저 받는다.

- 임무 목적이 두 개 이상으로 해석될 수 있음.
- 산출물 형식이 명확하지 않음.
- 외부 시스템 변경, 비용 발생, 데이터 삭제 가능성.
- 법률, 금융, 의료, 보안, 실작전성 판단 포함.

## 6. SOP-04 Multi-Agent Tasking

### 목적

여러 LLM 에이전트가 동시에 작업할 때 unity of command와 unity of effort를 유지한다.

### 적용 범위

- 리서치, 구현, 검증, 문서화를 병렬화할 때.
- 서로 다른 전문 역할이 필요한 작업.
- 한 에이전트의 판단을 다른 에이전트가 검토해야 할 때.

### 역할 구조

| 에이전트 | 군대식 대응 | 책임 |
| --- | --- | --- |
| Commander | 지휘관 | 목적, 우선순위, 승인 |
| Chief of Staff | 참모장 | 작업 흐름 통합 |
| S2 | 정보 | 리서치, 출처, 불확실성 |
| S3 | 작전 | 실행 계획, 태스크 분해 |
| S4 | 지속지원 | 도구, 토큰, 시간, 의존성 |
| S6 | 통신/지식 | 문서, 저장소, 컨텍스트 관리 |
| Red Team | 독립 검토 | 오류, 리스크, 반례 |

### 절차

1. Commander가 mission과 intent를 발행한다.
2. Chief of Staff가 작업을 role별로 분해한다.
3. 각 에이전트는 자기 임무를 backbrief한다.
4. S2는 근거와 불확실성을 먼저 제공한다.
5. S3는 실행 계획과 sequencing을 작성한다.
6. S4는 자원 제약과 병목을 보고한다.
7. 실행 중 CCIR가 발생하면 즉시 상위 판단으로 올린다.
8. Red Team은 최종 산출 전 독립 검토를 수행한다.
9. Chief of Staff가 결과를 하나의 산출물로 통합한다.

### 금지사항

- 하위 에이전트가 mission을 재정의하지 않는다.
- 리서치 에이전트가 실행 결정을 대신하지 않는다.
- Red Team이 직접 수정자가 되지 않는다.
- 각 에이전트가 서로 다른 용어 체계를 만들지 않는다.
- 출처 불확실성을 합의된 사실처럼 쓰지 않는다.

## 7. SOP-05 Code / Artifact Implementation

### 목적

LLM이 코드, 문서, 디자인, 데이터 파일을 변경할 때 변경 범위와 검증을 통제한다.

### 적용 범위

- 로컬 파일 편집.
- 코드 구현.
- 문서 생성.
- 테스트 실행.
- 빌드 또는 배포 준비.

### 절차

1. 현재 파일 구조를 확인한다.
2. 관련 파일을 읽고 기존 패턴을 파악한다.
3. 변경 전 의도와 범위를 짧게 보고한다.
4. 필요한 파일만 수정한다.
5. 수정 후 검색, 포맷, 테스트, 렌더링 중 필요한 검증을 수행한다.
6. 실패한 검증은 숨기지 말고 원인과 남은 위험을 기록한다.
7. 결과 요약과 다음 단계만 사용자에게 보고한다.

### 승인 기준

다음 변경은 실행 전 승인이 필요하다.

- 데이터 삭제.
- 외부 서비스 비용 발생.
- 보안 설정 변경.
- 공개 배포.
- 대규모 리팩터링.
- 사용자 변경사항을 되돌리는 작업.

## 8. SOP-06 Red Team Review

### 목적

산출물이 목적을 달성하는지, 근거가 충분한지, 위험이 숨겨져 있지 않은지 검토한다.

### 적용 범위

- 프레임워크 핵심 문서.
- 정책성 문서.
- 외부 공유 문서.
- 고위험 자동화.
- 근거 기반 답변.

### 검토 질문

1. mission과 intent가 산출물에 보존됐는가?
2. 출처가 주장과 직접 연결되는가?
3. 과장된 일반화가 있는가?
4. 검증 가능한 주장과 해석이 구분되는가?
5. 승인 없이 실행하면 안 되는 부분이 있는가?
6. 하위 에이전트가 오해할 용어가 있는가?
7. 실패 시 중단 기준이 명확한가?
8. AAR로 되돌릴 기록이 남는가?

### 산출물

- Findings.
- Severity.
- File/section reference.
- Recommended action.
- Residual risk.

## 9. SOP-07 Source and Evidence Management

### 목적

출처가 흩어지지 않게 하고, 문서의 주장을 나중에 검증 가능하게 만든다.

### 절차

1. 모든 출처는 `research-compendium.md`에 한 번 이상 기록한다.
2. 핵심 출처는 `source-map.md`에 군사 개념과 LLM 적용점을 연결한다.
3. 원문 직접 인용은 최소화하고 요약한다.
4. 불확실한 URL은 "확인 필요"로 표시한다.
5. 동일 출처가 여러 문서에서 쓰이면 source map을 기준점으로 삼는다.
6. 최신성이 중요한 문서는 확인 날짜를 남긴다.

### 출처 신뢰도 등급

| 등급 | 유형 | 사용 방식 |
| --- | --- | --- |
| A | 공식 교리, 정부/군 문서, 표준 | 핵심 근거 |
| B | 학술 논문, 공식 연구기관 보고서 | 보완 근거 |
| C | 현장 기사, 교육 자료, 핸드북 | 실무 적용 예시 |
| D | 블로그, 비공식 요약, 2차 인용 | 가설 또는 참고 |

## 10. SOP-08 FRAGO Change Management

### 목적

작업 중 명령 변경이 생겼을 때 전체 계획을 무너뜨리지 않고 필요한 부분만 갱신한다.

### 적용 범위

- 사용자가 중간에 요구사항을 바꿈.
- 새 리스크 발견.
- 출처가 기존 결론과 충돌.
- 작업 범위 추가 또는 축소.
- 시간 또는 자원 제약 변경.

### 절차

1. 기존 mission과 intent 중 보존되는 부분을 식별한다.
2. 변경된 항목을 situation, mission, execution, coordinating instructions로 나눈다.
3. 영향을 받는 문서, 코드, 에이전트를 식별한다.
4. 변경 후 우선순위를 다시 정한다.
5. 변경사항을 FRAGO 형식으로 기록한다.
6. 필요한 경우 backbrief를 다시 받는다.

### FRAGO 최소 양식

```text
Change:
Reason:
Affected tasks:
Unaffected intent:
New constraints:
Required confirmation:
```

## 11. SOP-09 AAR and Learning Loop

### 목적

작업이 끝난 뒤 성공/실패 원인을 기록하고 SOP, 프롬프트, authority matrix에 반영한다.

### 적용 범위

- 작업 완료 후.
- 실패 또는 재작업 발생 후.
- 사용자 피드백 수신 후.
- 반복 가능한 절차가 새로 발견됐을 때.

### AAR 질문

1. 의도한 결과는 무엇이었는가?
2. 실제 결과는 무엇인가?
3. 차이가 발생한 이유는 무엇인가?
4. 유지할 절차는 무엇인가?
5. 바꿀 절차는 무엇인가?
6. 다음 SOP 또는 프롬프트에 반영할 규칙은 무엇인가?

### 산출물

- AAR note.
- SOP update.
- prompt template update.
- risk register update.
- source map update.

## 12. SOP-10 Release / Publication Gate

### 목적

외부 공유 또는 장기 사용 전에 문서와 프레임워크를 점검한다.

### 체크리스트

- README에 링크가 있는가?
- 문서 목적이 첫 부분에 있는가?
- 각 핵심 주장에 근거가 있는가?
- 템플릿은 바로 복사해 사용할 수 있는가?
- 승인권과 보고선이 모호하지 않은가?
- 환각 방지 절차가 명시되어 있는가?
- AAR 반영 위치가 있는가?
- 남은 리스크가 기록되어 있는가?

## 13. 훈련 단계

군대의 training management를 LLM에 적용하면 SOP 숙달은 세 단계로 나뉜다.

| 단계 | 군대식 표현 | LLM 운용 |
| --- | --- | --- |
| Crawl | 절차를 따라 하는 단계 | 에이전트가 체크리스트 기반으로 수행 |
| Walk | 감독 아래 수행 | 일부 자율 판단 허용, backbrief 필수 |
| Run | 임무형 수행 | intent와 CCIR만 주고 자율 실행 |

중요한 점은 처음부터 자율성을 주지 않는 것이다. 에이전트가 SOP를 안정적으로 수행한다는 기록이 쌓일수록 권한을 넓힌다.
